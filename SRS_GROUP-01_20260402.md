## 1. Introduction

### 1.1 Purpose

This SRS document describes the functional and non-functional requirements for the AICAMS capstone project progress management system. It serves as the primary reference document for the development team and supervising instructors.

### 1.2 Scope

The system is a web platform directly integrated with Jira and GitHub. The main goal is to provide a centralized dashboard that helps supervisors track work progress, code commit volume, and student milestones without switching between multiple tools.

## 2. Overall Description

### 2.1 Product Perspective

The platform operates as an independent web application, communicating with the Atlassian ecosystem (Jira) and GitHub via REST API and Webhooks. The frontend runs on web browsers, and the backend is deployed in a Cloud environment.

### 2.2 Product Functions

The system provides the following major functions:

1. **Init Spring Boot 3 & Postgres** — Setup
2. **Authentication Module Deployment** — Detailed description for Authentication Epic
3. **Setup Sprint Management Module** — Management descriptions

1. Comprehensive dashboard for Supervisors.2. Automatic linking of Jira tasks with GitHub commits.3. Automated alert system when a repository has no activity for 7 days.4. Progress calculation (%) based on Epic/Story status.

### 2.3 Product Constraints

The system must operate within the rate limits of the free tier Jira and GitHub APIs. The project must be completed and deployed within a 15-week timeframe.

### 2.4 User Characteristics

Students: Have basic knowledge of Agile management and Git.Supervisors: Require an intuitive interface and quick operations to monitor multiple groups simultaneously without deep technical skills.

### 2.5 Assumptions and Dependencies

It is assumed that students are committed to using Jira for task management and GitHub for source code management. The system depends on the uptime of third-party APIs (Jira/GitHub).

### 2.6 Apportioning of Requirements

Core (Phase 1): Basic Jira/GitHub integration and Dashboard.Advanced (Phase 2): Advanced analytical reports and PDF export.

### 3.2 Functional Requirements

**ID**: REQ-FUNC-001
**Title**: Display Login User Interface
**Statement**: The system shall present a dedicated login user interface to users attempting to access protected resources or initiate a session.
**Rationale**: To provide a clear, intuitive, and consistent mechanism for users to authenticate themselves, ensuring adherence to design specifications and promoting a positive user experience.
**Acceptance Criteria**:
*   The login page shall display fields for "Username" and "Password".
*   The login page shall display a "Log In" button.
*   The login page shall display a link or option for "Forgot Password?".
*   The login page shall conform to the provided mockups and detailed design specifications for layout, styling, and branding.
*   Upon initial display, the username input field shall receive focus.
*   Input provided in the password field shall be masked.
*   The "Log In" button shall be disabled until both the username and password fields contain non-empty input.
*   If the username field is empty upon attempted submission, an error message "Username is required" shall be displayed adjacent to the field.
*   If the password field is empty upon attempted submission, an error message "Password is required" shall be displayed adjacent to the field.

---

**ID**: REQ-FUNC-002
**Title**: Authenticate User via Backend API
**Statement**: The system shall securely transmit user-provided credentials from the login user interface to the designated backend authentication API and process the API's response to establish or deny a user session.
**Rationale**: To securely verify user identities against the system's authentication service, thereby enabling authorized users to access application functionalities and protecting against unauthorized access.
**Acceptance Criteria**:
*   Upon clicking the "Log In" button, the system shall initiate an HTTP POST request to the `/api/login` endpoint (or other specified authentication endpoint).
*   The request body shall contain the entered username and password in a secure format (e.g., JSON).
*   During the transmission and processing of the authentication request, the "Log In" button shall display a loading indicator and remain disabled.
*   If the API returns a successful authentication response (e.g., HTTP 200 OK with a valid authentication token), the system shall redirect the user to the application's dashboard or the previously requested protected resource.
*   If the API returns an unsuccessful authentication response (e.g., HTTP 401 Unauthorized or HTTP 403 Forbidden), the system shall display a generic error message "Invalid username or password" on the login page.
*   The system shall handle network errors, API timeouts, or other HTTP error codes (e.g., 5xx) by displaying an appropriate error message such as "Unable to connect to authentication service. Please try again later."
*   Upon successful authentication, the system shall securely store the received authentication token (e.g., JWT) in a manner accessible for subsequent authenticated requests.

### 3.3 Quality of Service

#### 3.3.1 Performance

The API fetching Dashboard data for a Supervisor managing 10 groups must respond in under 2 seconds. The Webhook listener from GitHub must be processed within 500ms.

#### 3.3.2 Security

All communication via HTTPS. Passwords encrypted using BCrypt. Jira/GitHub tokens must be encrypted before being stored in the Database. Strict CORS configuration allowing client access.

#### 3.3.3 Reliability

Ensure data consistency when synchronizing from external systems. If the Jira/GitHub API fails, the system must fallback to the latest cached data and notify the user.

#### 3.3.4 Availability

Uptime of 99% throughout the semester. System maintenance window is scheduled for 2:00 AM - 4:00 AM.
